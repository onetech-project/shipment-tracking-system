## R-001: shadcn/ui Integration with Next.js 14 App Router

**Question**: How is shadcn/ui correctly initialised in a Next.js 14 App Router project?

**Decision**: Use the `shadcn` CLI (v4.x) with `components.json` targeting `src/components/ui/`
and Tailwind CSS v4.

**Rationale**:
- The official shadcn/ui docs describe exact Next.js 14 + App Router setup: run
  `npx shadcn@latest init` which generates `components.json`, installs `tailwindcss`,
  `class-variance-authority`, `tailwind-merge`, `lucide-react`, and all `@radix-ui/*`
  peer dependencies automatically.
- Components are **copied into the project** (not imported from an npm package), so they
  are fully customisable and add zero mandatory runtime overhead for unused parts.
- Tailwind CSS v4 (current: 4.2.x) ships with a native PostCSS plugin; no separate
  `tailwind.config.js` is required when using the v4 CSS-first config strategy. However,
  shadcn/ui still generates a `tailwind.config.ts` for component configuration — both modes
  are supported.
- With Next.js 14 App Router, `globals.css` must import Tailwind via `@import "tailwindcss"`
  (Tailwind v4 syntax) plus the shadcn CSS variable block.

**Alternatives Considered**:
- `shadcn-ui` (v0.x legacy package) — rejected: deprecated, does not support Tailwind v4 or
  Next.js 14 RSC.
- Manual Radix UI + custom Tailwind — rejected: significantly more boilerplate; shadcn/ui
  already provides the exact component list required by FR-003.

**Implementation Steps**:
```bash
cd apps/frontend
npx shadcn@latest init         # generates components.json + installs deps
npx shadcn@latest add button input label card badge table dialog sheet \
    dropdown-menu form separator avatar
```

**References**:
- https://ui.shadcn.com/docs/installation/next
- https://ui.shadcn.com/docs/components

## R-002: Tailwind CSS v4 Configuration Strategy

**Question**: Should we use Tailwind v3 (config-file-based) or v4 (CSS-first)?

**Decision**: Use **Tailwind CSS v4** with its CSS-first configuration (`@import "tailwindcss"`
in `globals.css`) combined with the `@theme` block for design tokens. Retain a minimal
`tailwind.config.ts` for shadcn/ui content-path configuration.

**Rationale**:
- Tailwind v4 (current: 4.2.x) delivers native CSS variables for all tokens, eliminating
  the need for `tailwind.config.js` `theme.extend` boilerplate for most design decisions.
- CSS custom properties (CSS variables) defined in `:root {}` inside `globals.css` fulfil
  FR-015: the entire color scheme can be updated by editing one file.
- shadcn/ui generates its own CSS variable names (e.g., `--primary`, `--background`); Tailwind
  v4's `@theme inline` directive maps these to Tailwind utility classes automatically.
- `motion-safe` and `motion-reduce` Tailwind variants are available in v4 and fulfil FR-007 /
  SC-007 with zero JavaScript — transitions are simply declared as
  `motion-safe:transition-shadow motion-reduce:transition-none`.

**Alternatives Considered**:
- Tailwind v3 with `tailwind.config.js` `extend` — rejected: v4 is the current release and
  avoids a future migration; v4 also provides better CSS variable integration.
- Custom CSS animation library (e.g., Framer Motion) — rejected: YAGNI; Tailwind transition
  utilities fully satisfy FR-005 / FR-006 at zero extra bundle cost.

**CSS Variable Contract (from shadcn/ui init)**:
```css

  --card: 0 0% 100%;
  --card-foreground: 222.2 84% 4.9%;

  --radius: 0.5rem;
  --sidebar-bg: 222.2 47.4% 11.2%;    /* slate-900 — existing sidebar colour */
  --sidebar-fg: 210 40% 98%;
}
```

## R-003: Responsive Sidebar — Sheet vs. Drawer vs. Custom

**Question**: What is the recommended pattern for a collapsible mobile sidebar in Next.js +
shadcn/ui?

**Decision**: Use **shadcn/ui Sheet** component (built on Radix UI Dialog) for the mobile
slide-in drawer. The desktop sidebar is a static `<aside>` rendered only at `lg:` breakpoint.
A single `useSidebar` hook drives the open/close state.

**Rationale**:
- Sheet provides the accessible focus-trap, escape-key close, and smooth slide animation
  required by FR-004 / User Story 1 out of the box.
- Using a Tailwind breakpoint split (`hidden lg:block` for sidebar; `lg:hidden` for hamburger
  button + Sheet trigger) is the simplest approach with zero JavaScript for the layout switch.
- Sheet animates via `data-[state=open]` CSS — compatible with `motion-reduce` override.
- Using a single `useSidebar` context hook (local to `components/layout/`) keeps state
  management minimal and avoids prop-drilling across the layout tree.

**Mobile Breakpoint**: `1024 px` (Tailwind `lg:`) — sidebar visible on `≥ 1024 px`,
hamburger on `< 1024 px`. This matches FR-004 and User Story 1 acceptance scenarios.

**Alternatives Considered**:
- Custom CSS drawer with `transform: translateX` — rejected: requires manual a11y (focus trap,
  escape handling); Sheet already provides this.
- `vaul` drawer library — rejected: additional dependency; YAGNI since Sheet covers the use case.
- Headless UI Disclosure — rejected: not part of shadcn/ui stack; introduces inconsistency.

## R-004: Animations — Tailwind vs. Framer Motion

**Question**: Are Tailwind CSS transition utilities sufficient to meet FR-005, FR-006, FR-007?

**Decision**: **Yes — Tailwind transition utilities only.** No additional animation library is
needed.

**Rationale**:
- Hover/focus effects (FR-005): `transition-all duration-150 hover:shadow-md hover:scale-[1.02]`
  on buttons and cards; `focus:ring-2 focus:ring-primary` on inputs.
- Modal animation (FR-006): shadcn/ui Dialog ships with built-in Radix UI entrance/exit
  animations expressed as Tailwind `data-[state=open]:animate-in data-[state=closed]:animate-out`
  plus `fade-in-0 zoom-in-95` utilities — this matches the spec's "fade + scale" requirement.
- Reduced motion (FR-007 / SC-007): wrap all non-essential transitions in `motion-safe:` variant,
  e.g. `motion-safe:transition-shadow motion-safe:hover:shadow-md`. When `prefers-reduced-motion:
  reduce` is active, Tailwind omits these rules automatically — no JavaScript required.
- 150 ms default (`duration-150`) meets the SC-002 "within 200 ms" requirement.

**Alternatives Considered**:
- Framer Motion — rejected: overkill for button hover effects; adds ~60 KB to the bundle.
- CSS keyframe animations written by hand — rejected: shadcn/ui already ships suitable
  keyframes; duplicating them violates DRY.

## R-005: Data Table Pattern — shadcn/ui Table vs. TanStack Table

**Question**: FR-010 requires horizontally scrollable tables with empty states. Should we use
shadcn/ui's `Table` primitives directly or wrap TanStack Table?

**Decision**: Use **shadcn/ui `Table` primitives** wrapped in a custom `DataTable` shared
component. Do **not** introduce TanStack Table in this revamp.

**Rationale**:
- All tables in the application have simple, static column sets — no sorting, filtering, or
  virtualisation is required at this time (YAGNI).
- The `DataTable` wrapper adds only: a `overflow-x-auto` scroll container, a consistent
  `thead` style, row `hover:bg-muted/50 transition-colors` effect, and an empty-state slot.
- This satisfies FR-010 (horizontal scroll + empty state) and FR-005 (row hover) with a single,
  simple component.

**Alternatives Considered**:
- TanStack Table — rejected: significant added complexity and bundle size; not needed for
  current requirements (YAGNI, KISS).
- Each page with its own table markup — rejected: violates DRY, FR-008 (shared primitives).

## R-007: data-testid Preservation Strategy

**Question**: How do we ensure all 25+ existing `data-testid` attributes survive the revamp?

**Decision**: Maintain a **testid inventory** (listed in contracts/shared-components.md) and
treat each `data-testid` as a contract that must be preserved on the revamped element, regardless
of the surrounding markup change.

**Rules**:
1. Every `data-testid` present in `apps/frontend/src/` must appear on the same logical element
   in the revamped version.
2. shadcn/ui wrapper components (e.g., `<Button>`) accept standard HTML attributes including
   `data-testid` via the `...props` spread — no extra work needed.
3. The `DataTable` wrapper must forward `data-testid` props to the outer element.
4. New Playwright tests in `e2e/ui-responsive/` use `data-testid` where they target existing
   elements, and use Tailwind class or ARIA role selectors for new responsive-only elements
   (e.g., `[aria-label="Open menu"]` for the hamburger button).

**Existing testid inventory**:
- `file-type-error`, `upload-submit` — PdfUploader
- `import-status`, `rows-imported` — ImportStatus
- `conflict-review`, `conflict-row`, `conflict-action-overwrite`, `conflict-action-skip`,
  `resolve-conflicts-submit` — ConflictReview
- `history-empty`, `history-row` — UploadHistory
- `start-scanner`, `permission-prompt`, `permission-denied`, `no-camera`, `camera-in-use`,
  `scanner-status`, `shipment-not-found`, `invalid-qr-format`, `scan-error` — QrScanner
- `shipment-detail` — ShipmentDetail
- `linehaul-detail`, `linehaul-trip-header`, `scan-again-button` — LinehaulDetail

## R-008: Login Page Layout Pattern

**Question**: What is the standard shadcn/ui pattern for a centered login card?

**Decision**: Wrap the login form in a shadcn `Card` with `CardHeader`, `CardContent`, and
`CardFooter`. Use shadcn `Form` (which wraps `react-hook-form`) with `FormField`, `FormItem`,
`FormLabel`, `FormControl`, `FormMessage` for each field. Center with `flex min-h-screen
items-center justify-center bg-muted/40` on the page root.

**Rationale**: This is the canonical shadcn/ui pattern; it directly uses the components already
added by `shadcn add form input label`. The existing `react-hook-form` + `zod` logic is
preserved unchanged inside the `onSubmit` handler — only the JSX markup changes.

## R-009: "prefer-reduced-motion" Implementation

**Question**: Does the Tailwind `motion-safe` / `motion-reduce` approach require any JavaScript?

**Decision**: **No JavaScript required.** Use Tailwind's `motion-safe:` variant prefix on all
non-essential animation classes.

**Pattern**:
```tsx
// Button hover shadow — suppressed when reduce-motion active
<Button className="motion-safe:transition-shadow motion-safe:hover:shadow-md">

// Modal animation — shadcn Dialog uses data-[state] attributes
// wrap the animate-in/out classes:
<DialogContent className="motion-safe:data-[state=open]:animate-in
                          motion-safe:data-[state=open]:fade-in-0
                          motion-safe:data-[state=open]:zoom-in-95">
```

**Rationale**: Tailwind's `motion-safe:` variant maps directly to `@media (prefers-reduced-motion: no-preference)`.
Classes without the prefix always apply. Classes with the prefix apply only when the user has
not requested reduced motion. No `useReducedMotion` hook or JavaScript event listener is needed.

## R-010: Next.js 14 App Router + Tailwind CSS — Known Gotchas

**Decision**: Apply the following mitigations proactively:

| Gotcha | Mitigation |
|--------|-----------|
| Tailwind purges Server Component class names when they are built dynamically | Only use static Tailwind class strings (no string interpolation); use `cn()` helper from `tailwind-merge` for conditional joining |
| `"use client"` boundary needed for Sheet/Dialog state | Layout components that use `useState` for mobile drawer open state must be Client Components; the overall `layout.tsx` can still be a Server Component if it delegates to a `DashboardShell` Client Component |
| shadcn/ui `Form` requires `react-hook-form` `<FormProvider>` | Already using `react-hook-form`; wrap existing form components in the shadcn `Form` wrapper (which is just `<FormProvider>`) — no changes to validation logic |
| Tailwind v4 content paths | Set `content: ["./src/**/*.{ts,tsx}"]` in `tailwind.config.ts` |
